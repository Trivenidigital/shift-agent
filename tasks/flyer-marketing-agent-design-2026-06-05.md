# Flyer Studio → Marketing Agent — Architecture Design (2026-06-05)

**Drift-check tag:** extends-Hermes
**Status:** DESIGN for operator sign-off — **no code until approved**.
**Provenance:** senior-architect assessment → operator-approved synthesis. **Rev 3.** Rev 2 folded in: Creative Director is slice-1, overlay owns all required facts, staged routing, Hermes-primary-not-decorative. Rev 3 tightens the skill contract (skill returns *structure*, not fact authority; required fact set is deterministic; one creative brain not many), adds the QA send-path migration point and the Hermes-research support. **Rev 3.1** adds the exact-source-edit carve-out + the raw-request-span materialization lifecycle. Supersedes the "bare trunk"; the bare trunk is an input to slice 1's single render path, not a parallel renderer.

---

## 🎯 North Star — the target product

Flyer Studio becomes the **marketing brain for an ethnic SMB.** The owner texts a vague line — *"diwali sweets offer"* — and the agent already knows it's Lakshmi's Kitchen: it pulls their bestsellers, knows Diwali's date for their region, composes an on-brand flyer in *their* colors and language (English + Telugu), grounds every price/name/phone as fact (never invented), shows the owner a draft to approve, and — once approved — turns it into a **measurable campaign** with an order link, broadcast to their shopper list. And it doesn't only wait to be asked: it **nudges proactively** off the festival/seasonal calendar — *"Diwali's in 9 days — want a sweets-box flyer?"* The flyer is one output of that agent; **the agent is the product.**

Everything below serves that destination. **Slice 1 builds the agent's reliable foundation**; slices 2+ add the **proactivity** (festival calendar, #2) and the **revenue loop** (commerce, #3). The order is deliberate — an unreliable marketing agent is worse than none — but the destination, not the foundation, is the point.

## 1. Problem statement

Live runtime, last ~week: **1,272 `visual_qa_failed` vs 19 `outbound_sent`**, 1,256 source-edit SLA alerts, 203 `closed_no_send`, code running from three divergent staging trees with `flyer.enabled=false`. The product fails far more than its architecture requires — self-inflicted: a QA gate that rejects its own correct output; five contradictory render pivots in 20 days; a regex routing layer; **creative judgment trapped in Python prompt-builders that bypass Hermes**; deployment sprawl.

The reframe: become the marketing agent above. **Most parts already exist** (profile, brand kit, locked facts, a firewall-aware creative planner, commerce primitives, an acceptance oracle) — dormant and disconnected. This is **assembly + consolidation, not new construction** — the guardrail against pivot #6.

## 2. The invariant (non-negotiable, carried into every slice)

1. **Locked facts are never model-authored.** All required visible facts — business identity, campaign title/headline, schedule/date, item names, prices, CTA/footer/contact — are *placed* deterministically from the knowledge model, never generated by the image model. A wrong customer-facing fact must be *structurally impossible*.
2. **Owner approval gates anything broadcast to shoppers.** "Customer" for QA = the owner/requester, who reviews drafts. End-shoppers see only owner-approved output.
3. **Owner review replaces *subjective* QA — not deterministic safety.** The system still blocks wrong-business, unauthorized/ambiguous sender, blank/corrupt/unsafe image, and impossible source-edit cases *before the owner sees a draft*. What moves to the owner is taste/layout judgment, never safety or ownership.
4. **Offer structure is preserved exactly.** Combo/package/deal → combo cards; menu → menus; event → event layout; source-edit → source hierarchy. Never expand a stated offer into extra items unless the customer explicitly asks for suggestions.

## 3. Architectural correction — Hermes skills/plugins are primary, not decorative

**Hermes-first here means the skill/plugin *is* the implementation, not a doc that wraps Python.** Load-bearing because the current tree does the opposite: creative judgment lives in `render.py:_image_prompt` + `campaign_scene_prompts.py` (Python keyword buckets), while `flyer_dispatcher/SKILL.md` routes normal requests around the skill.

Any judgment-heavy behavior is expressed **first** as a Hermes skill/plugin capability: creative direction, occasion/theme interpretation, request intent, layout strategy, prompt simplification, revision understanding, "what not to add." If no skill exists, **create/update the SKILL.md first**, then add only the minimal glue to make production invoke it.

**Custom Python is allowed only for:** sender/customer/business-fact resolution · locked-fact extraction and validation · source ownership / safety / danger gates · deterministic overlay of customer-facing facts · state/audit/retries/delivery/tests · calling/delegating to the Hermes skill/plugin path.

Do **not** encode Memorial Day / Diwali / Ugadi / combo-deal-menu creative judgment as expanding Python heuristics — `campaign_scene_prompts.py` is exactly this anti-pattern, and Memorial Day isn't even in its keyword set, which is *why* it rendered generic Indian styling. **Python validates the skill's output; it does not become the creative brain.**

**Skill vs plugin (Hermes definitions):** a **skill** is the right abstraction for task-specific *behavior/judgment* loaded on demand (the creative brain); a **plugin** is the right abstraction for callable *tools/hooks/integrations* and can bundle skills + register tools (the unavoidable wiring + safety envelope). The creative director is a **per-request FlyerBrief builder (a skill)** — **not** a fork/wrapper of Hermes core `prompt_builder.py`, which is system-prompt assembly infrastructure. Use Hermes prompt layering + skill disclosure; do not extend core prompt assembly.

**Wiring is mandatory.** A SKILL.md update is inert unless the active production path consults it. The implementation plan MUST name the wiring point where `cf-router` / `generate-flyer-concepts` / render-prompt construction delegates to the skill (a plugin-backed structured-LLM call). Division: **Hermes skill = creative brain; cf-router plugin = safety envelope** (the Python allowlist above).

## 4. The spine — Business Knowledge Model

One first-class per-business read-model every output reads from — a **unifying layer over objects that already exist**, plus pre-provisioned seams:

| Dimension | Status on origin/main | Spine action |
|---|---|---|
| Brand memory | `FlyerBrandKit` (schemas.py:1882), `FlyerBrandAsset` (1053) **exist** | **first-class** — every flyer unmistakably *them* |
| Business profile / hard facts | `FlyerCustomerProfile` (1142), `profile_locked_facts` (facts.py:161), `FlyerLockedFact` (1686) **exist** | canonical fact source for the overlay |
| Language / bilingual | `FlyerCustomerProfile.languages` + Telugu overlay fonts (`render.py`) **exist** | **first-class** — English + Telugu by default |
| Customer store | `FlyerCustomerStore` (1327) **exists** | per-VPS JSON home (no new storage) |
| Menu/catalog + CTA/order-link | commerce **primitives** exist (`src/platform/commerce/`); **flyer catalog identity model does NOT** | **net-new seam schema** now (dormant), backed by primitives in slice 3 |
| Festival/region profile | none | **net-new**, small — powers the proactive calendar (#2) |
| Campaign history | none | **net-new**, slice 3 (attribution) |

Storage stays JSON-on-disk + `safe_io` (no DB). The spine is a typed read-model + seam fields — not a framework.

## 5. Slice 1 — the agent's reliable foundation

The RC consolidation on the spine — **the foundation the agent stands on, not a separate renderer project.** The only slice that ships customer-facing behavior first.

**Render — one path, brain-and-envelope (the skill proposes; deterministic code decides truth):**
1. **Hermes Creative Director skill** returns **one structured `FlyerBrief`** + a clean textless-background prompt. It may express `request_intent` (combo_offer / menu / event / source_edit), `offer_structure`, `visual_direction` (theme/palette/motifs/`visual_subjects`), `layout_strategy`, grouping/emphasis/priority, and `must_not_add`. **Commercial values appear only as fact-IDs or raw-request spans *with provenance* — never free-floating "truth."** This replaces the Python `render.py:_image_prompt` + `campaign_scene_prompts.py`. It is **one** creative brain — *not* a constellation of slice-1 handler-skills (no `ComboIntentHandler` / `PriceRevisionHandler` / `SourceAdaptationHandler` / `AdvisoryCritic`); that would recreate the fragmented router in skill form.
2. **Deterministic fact authority (the firewall).** Code — not the skill — computes the **required visible fact set** from locked facts + `request_intent`. It validates that every commercial value the skill referenced maps to a locked fact **or** a raw-request span. The skill may only influence grouping/emphasis/layout-priority of those facts, **not which facts are required.** A raw-request span is **validation evidence only** — deterministic code **materializes** a validated span into a `FlyerLockedFact(source="customer_text")` before anything renders it; the overlay never renders directly from a skill-provided span.
3. **Model paints a textless background** from the background prompt (visual subjects + motifs, **no words**).
4. **Deterministic overlay renders `required_fact_ids ∩ locked_facts`** — never the skill's list alone. (Skill can't *invent* a fact (step 2 validator) and can't *omit* a required one (step 2 required-set). Both truth-gap directions closed.) Combo cards + prices included. No heavy HTML/Playwright compositor.
- **All *new* and *source-inspired* render paths collapse into this one — but exact source-preserving edits do NOT.** A request to edit uploaded artwork ("remove the extra 08:00", "change this date/name") uses the verified source-edit path, or falls to the **danger/manual queue when an exact edit is impossible**; it is never silently recreated as a textless-background flyer and mislabeled an edit. The textless-background path is for new / source-*inspired* work only.
- **QA = danger gate only — with a named migration point.** Today `generate-flyer-concepts` blocks on `run_visual_qa` → `status="manual_edit_required"`, `reason_code="visual_qa_failed"` (:585). Slice 1 **rewrites the QA severity/action mapping** so only danger classes (wrong-business/ownership, sender, blank/corrupt/unsafe image, impossible source-edit) block before owner preview; OCR "missing fact/name/price" becomes **advisory** once the deterministic overlay owns those facts.
- **Owner loop is the normal path.** Draft → "Reply with changes, or APPROVE." Changes → revision returns to the **active draft**; APPROVE → finalize. Vague → **clarify** (one question), never a hard fail.
- **Routing (RC3) — staged, not a hard swap.** Promote the existing Hermes intent path to **shadow**; keep the regex as **fallback** until the golden set is green and replay + live shadow-disagreement evidence clears (repo routing-reliability convention — `dispatcher-accuracy-report` shadow comparison); only then delete the `cf-router/actions.py` regex arms.
- **Make the skill unavoidable, or it's inert.** `flyer_dispatcher/SKILL.md` routes normal requests around the skill; `flyer_generation/SKILL.md` currently tells the model to render exact facts (integrated-poster guidance — rewrite to creative-director guidance). Slice 1 wires `generate-flyer-concepts` / cf-router to **call** the skill. cf-router stays the safety envelope, no longer the creative judge.
- **Residual guard:** detect (or owner-review for) **model-leaked text inside the textless background.**
- **Convergence gate:** an acceptance-oracle **send-rate** on a fixed golden set (~15–20 real briefs incl. a dense 30-item list + a combo like Memorial Day). Metric to be defined (§11). "Done" = green for N days; no declaring done by survey.

## 6. Differentiators — seams now, built after slice 1 is green

- **#1 — creative intelligence, split across slices.** The **art-direction + structure-preservation** half *is* the Creative Director skill, in **slice 1** (§5). The **content-suggestion** half (suggesting items not listed) is `creative_planner` — verified item-only ("emits `item:*:name` candidates", creative_planner.py:14) — and stays **slice 2** behind the firewall (`materialize_inferred`:218 + `promote_inferred_to_confirmed`:224, owner-confirmed). **Slice 1 faithfully renders what's asked (themed); slice 2 suggests what isn't.** Kills the 12-items/`FAMOUS_ITEM_SETS` class (facts.py:299).
- **#2 — proactive festival/seasonal calendar.** Knows the ethnic calendar with regional splits and nudges first. **Net-new but small** — a regional calendar *data* file + a scheduler tick on the daily-brief cadence. (Occasion *data* is the calendar; occasion *creative interpretation* stays in the skill, never Python.)
- **#3 — commerce loop.** Flyer → CTA/order-link → measured orders, broadcast only after owner approval. **Pre-provision now:** spine catalog records carry CTA/order-link/price-as-commerce-object fields *in slice 1* (net-new seam schema, dormant); slice 3 wires into `src/platform/commerce/`.

## 7. Upgrade trigger (the deferred compositor)

The heavy HTML/Playwright compositor is **deferred, not rejected.** Trigger = golden-set send-rate plateauing **and** dense-layout failures (the 30-item grocery flyer the Pillow overlay can't lay out). The Parity-harness idea lives here — the measurement that decides *when* to upgrade, not the truth mechanism.

**External Hermes ecosystem check (Skills Hub + awesome-Hermes), with architecture support:**

- Hermes **architecture**: `prompt_builder.py` is core prompt *assembly*, not product-specific brief building → build the brief as a **skill**, don't fork core.

- **Market research:** no flyer-specific public Hermes skill replaces the in-tree creative director → **build/extend in-tree `flyer_generation`.** Image/design skills (FLUX / open-design / MeiGen / detect-skill) are **not** a slice-1 replacement for the deterministic fact-overlay invariant.
- **Existing main-vps substrate skills** (google-workspace, maps, airtable, notion, ocr-and-documents, email/himalaya, `mcp/native-mcp`) are **substrate, not the flyer brain.**
- **For other agents:** prefer MCP/vendor connectors first for QBO, Stripe, Square, PayPal, Razorpay, DocuSign; custom API work only after connector review.

## 10. Non-goals (explicit, to prevent pivot #6)

- **No new parallel renderer.** Slice 1 *removes* render paths.
- **No creative judgment in Python.** No new/expanding occasion/festival/intent heuristics; Python validates skill output only.
- **One creative brain, not many.** No fragmented per-task slice-1 skills; no forking/wrapping Hermes `prompt_builder.py`.
- **Skill is not fact-authoritative.** Required facts are deterministic; the skill references facts by ID/span only.
- **No Stripe/payment writes, no broadcast in slice 1.** Commerce seam-only until slice 3, owner-gated.
- **No pro-model spend until the golden set is green.** No heavy compositor until the upgrade trigger fires.
