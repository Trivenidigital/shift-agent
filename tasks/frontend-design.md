# Owner Cockpit — Design v1.1

**Status:** design, parallel-reviewed (3 agents); below corrections applied during build
**Incorporates:** plan v1 + all critical/high findings from 3-agent plan review + 3-agent design review

## Design v1.1 — corrections applied (during build)

From design-review feedback, the following changes are made directly in code (not re-spec'd in the prose below):

1. **Stack flip — Tailwind 4 + shadcn 4** (latest, stable 2026). Drops `tailwind.config.js`, uses CSS-first config. shadcn@4 install.
2. **QR over SSE = raw Baileys data string**, not PNG-base64. Frontend `qrcode.react` consumes it directly.
3. **STATUS_BADGE compile-time exhaustive check** via `satisfies Record<ProposalStatus, …>`.
4. **Section state mirrored to `?s=…` URLSearchParam** (back-button + Pushover deep links).
5. **`roster_lock_and_load` switched to caller-driven `commit()`** — no write on exception, re-validate after mutation.
6. **SSE pair lifecycle**: in-memory `PairSession` registry + reaper task (60s idle → SIGTERM bridge); `/run/shift-agent/cockpit-pair-{sid}.txt` mode 0600 (not `/tmp`); POST `/repair` returns 409 if active session exists.
7. **CLI is source of truth for proposal transitions** — backend does NOT pre-validate, surfaces CLI exit + stderr as 4xx.
8. **`owner.phone` added to MFA-gated config fields** (account-takeover protection).
9. **Subprocess args**: type checked + content sanitized for `--` injection; `--` terminator before user-supplied positional args.
10. **`/health` public**: returns `{ok}` only; full payload in `/dashboard` (auth-gated). Plan §6 contradiction noted but design is authoritative.
11. **Audit log rotation**: backend uses `logging.handlers.WatchedFileHandler`; logrotate config with `nocreate` + `prerotate chattr -a` + `postrotate chattr +a`.
12. **JWT secret**: 256-bit min, generated by `secrets.token_hex(32)` at first deploy, written to `/opt/shift-agent/.env`. Rotation == manual (.env edit + restart) — known Phase 3 gap.
13. **State writes use `safe_io.load_model` / `dump_model`** (not raw read/write); modes preserved or default 0o600 (no 0o644 leak of phones).
14. **AT row corrected: 11 ProposalStatus variants** (was 10).
15. **OTP timing equalization**: hmac.compare_digest + minimum 200ms wall-time floor on the verify handler (asyncio sleep delta).
16. **Bridge perf debt acknowledged**: cockpit calls existing CLIs via `subprocess`. ~200ms tax accepted for v1; Phase 3 imports as Python.

## 2. Backend — FastAPI

### 2.1 Auth (incorporating security review)

```python
# app/auth.py
import hmac, secrets, time
from datetime import datetime, timedelta, timezone

OTP_TTL_SECONDS = 300            # 5 min
OTP_MAX_ATTEMPTS = 5             # lockout per token (security gap fix)
OTP_REQUEST_PER_IP = (3, 900)    # 3 per 15 min
OTP_REQUEST_PER_OWNER = (5, 3600)  # 5 per hour
JWT_TTL_HOURS = 24
JWT_ALGO = "HS256"

def constant_time_eq(a: str, b: str) -> bool:
    return hmac.compare_digest(a.encode(), b.encode())  # security: equal-time
```

Storage: OTP state lives in `/opt/shift-agent/state/cockpit-otp.json` (single owner, single OTP active at a time). Schema:

```json
{
  "token": "32-byte-hex",        // server-generated, returned to client
  "code": "123456",              // 6 digits, hmac-comparable
  "issued_to": "+17329837841",   // owner.phone from config
  "expires_at": "2026-04-26T...",
  "verify_attempts": 0,
  "ip_issued": "1.2.3.4",
  "ua_issued": "Mozilla/5.0..."
}
```

Verify flow:
1. Client `POST /auth/verify-otp { token, code }`
2. Server reads cockpit-otp.json under flock
3. If `token` not present → return 401 with **same wall-time** as a wrong-code response (sleep to equalize)
4. If `verify_attempts >= 5` → invalidate token, return 401 "too many attempts; request new OTP"
5. `hmac.compare_digest(stored.code, code)` — if false, increment attempts, return 401
6. If true → mint JWT, set HttpOnly cookie, delete OTP record
7. JWT claims: `{sub: owner.phone, iat, exp, jti}`

OTP delivery: Pushover with `priority=2 + emergency`, custom sound. Code is 6 digits, generated `secrets.choice(string.digits)`.

### 2.2 Subprocess wrapper (RCE defense)

```python
# app/shell.py
ALLOWED_BINS = frozenset({
    "/usr/local/bin/identify-sender",
    "/usr/local/bin/create-proposal",
    "/usr/local/bin/update-proposal-status",
    "/usr/local/bin/send-coverage-message",
    "/usr/local/bin/render-coverage-template",
    "/usr/local/bin/log-decision",
    "/usr/local/bin/shift-agent-disable",
    "/usr/local/bin/shift-agent-enable",
    "/usr/local/bin/shift-agent-smoke-test.sh",
})

def run_cli(binary: str, args: list[str], timeout: float = 30) -> CompletedProcess:
    if binary not in ALLOWED_BINS:
        raise ValueError(f"binary not in allowlist: {binary}")
    if any(not isinstance(a, str) for a in args):
        raise TypeError("args must be all strings")
    return subprocess.run(
        [binary, *args],          # list, not string
        shell=False,              # explicit
        capture_output=True,
        text=True,
        timeout=timeout,
        check=False,
    )
```

Every router uses `run_cli`. No raw `subprocess` anywhere else.

### 2.3 State coordination (single-flock-per-file)

Roster + schedule share `roster.json`. Backend exposes them as separate REST resources but writes go through a single helper that:
1. Acquires `safe_io.flock` on `roster.json`
2. Loads → validates against `Roster` Pydantic model
3. Applies the patch in-memory
4. Re-validates the full Roster (referential integrity)
5. `safe_io.atomic_write_text(roster.json, new_content, mode=0o644)`

```python
# app/state.py
@contextmanager
def roster_lock_and_load() -> Iterator[Roster]:
    with safe_io.flock(ROSTER_LOCK_PATH):
        roster = Roster.model_validate_json(ROSTER_PATH.read_text())
        yield roster
        # Caller mutates roster, then writes:
        safe_io.atomic_write_text(ROSTER_PATH, roster.model_dump_json(indent=2), mode=0o644)
```

Backend usage:

```python
# routers/roster.py
@router.patch("/employee/{employee_id}")
def patch_employee(employee_id: str, patch: EmployeePatch, _=Depends(require_auth)):
    with roster_lock_and_load() as roster:
        emp = next((e for e in roster.employees if e.id == employee_id), None)
        if not emp:
            raise HTTPException(404)
        if patch.phone and patch.phone != emp.phone:
            emp.phone_history = (emp.phone_history or []) + [
                PhoneHistoryEntry(phone=emp.phone, effective_from=emp.phone_set_ts, effective_to=now())
            ]
            emp.phone = patch.phone
            emp.phone_set_ts = now()
        # apply other fields
        ...
    audit.log("roster.employee.patch", {employee_id, patched_fields})
```

### 2.4 Pending proposals — full discriminated union

`schemas.py` Proposal union has 10 status variants. Cockpit must surface all. Backend:

```python
# routers/pending.py
@router.get("", response_model=list[ProposalView])
def list_pending(
    include_terminal: bool = False,
    since: datetime | None = None,
    _=Depends(require_auth),
):
    with safe_io.flock(PENDING_LOCK):
        store = PendingStore.model_validate_json(PENDING_PATH.read_text())
    out = []
    for p in store.proposals.values():
        if not include_terminal and p.status in TERMINAL_STATUSES:
            continue
        out.append(ProposalView.from_proposal(p))
    return sorted(out, key=lambda x: x.last_updated_ts, reverse=True)

@router.post("/{pid}/cancel")
def cancel(pid: str, body: CancelBody, ip: str = Depends(client_ip), _=Depends(require_auth)):
    # Run the existing CLI under the hood
    r = run_cli("/usr/local/bin/update-proposal-status",
                [pid, "cancelled", "--cause", body.reason, "--actor", "owner_cockpit"])
    if r.returncode != 0:
        raise HTTPException(400, r.stderr)
    audit.log("pending.cancel", {"proposal_id": pid, "reason": body.reason, "ip": ip})
    return {"ok": True}
```

`ProposalView` has status-specific badge colors and which actions are valid (Cancel only allowed if `is_legal_transition(status, "cancelled")`).

### 2.5 WhatsApp re-pair — SSE flow (resumable)

```
POST /whatsapp/repair                       → 200 {session_id, expires_at}
                                             (also kicks off the bridge --pair-only,
                                              writes to /tmp/cockpit-pair-{sid}.txt)
GET /whatsapp/repair/{sid}/stream           → text/event-stream
   event: qr           data: <base64-qr-png>
   event: connected    data: {me: "+17329837841", paired_at: "..."}
   event: error        data: {message: "..."}
   event: complete     data: {me, self_chat_jid, registered}

POST /whatsapp/repair/{sid}/cancel          → kill bridge process
```

Backend writes the QR text → reads via inotify or polling → emits as SSE events. Client renders QR using `qrcode.react` (50KB). Reconnection: client reconnects to the same `sid` and resumes.

**Post-pair self_chat_jid update**: After `complete`, backend reads `/root/.hermes/whatsapp/session/creds.json`, extracts `me.id` (e.g., `17329837841:54@s.whatsapp.net`), constructs `self_chat_jid = "17329837841@s.whatsapp.net"`, and PATCHes `config.yaml` under flock. Audit logged.

## 8. Threats deferred to Phase 3 (explicit)

Items the security-reviewer flagged that we accept-the-risk for v1 because they don't materially raise compromise probability for a single-user trusted owner:

- **TOTP fallback** — accept Pushover-only. Mitigation: documented SSH escape-hatch in runbook §6 (`sudo -u shift-agent /usr/local/bin/shift-agent-enable`).
- **Remote audit shipping** — accept on-disk only. `chattr +a` + nightly backup to `/opt/shift-agent/backups/` is the v1 tamper bound.
