## Design constraints

1. **Test-only PR.** Zero changes to production code, schemas, or SKILL prompts. Pure pytest additions.

2. **Match existing test patterns:** subprocess-via-`_run_create` wrapper, `bridge_server` + `BridgeStub` fixtures, `env_dir` for isolated state, `_bridge_post_text(BridgeStub)` helper from PR #27.

3. **Each case → one dedicated test function** named `test_c<NN>_<short_description>`. Doc-spec assertions appear verbatim in test docstring (so v3.1 doc and test code are bidirectionally traceable).

4. **C16 + C17** test `apply-catering-owner-decision._load_menu_filtered` — different surface than the lead-creation tests. Use existing `_run_apply` wrapper for these. Need to seed a `catering-menu.json` fixture with veg/non-veg items.

5. **C04 implicitly tests `lookup_prior_leads_by_phone`** — already tested in `test_lookup_prior_leads.py` for various phone variants. C04's contribution: confirm `create-catering-lead` does NOT trigger auto-linking based on `notes` content (privacy invariant). One assertion that the unknown phone's lead is NOT linked to Priya's existing leads.

6. **No mock LLM.** All cases test deterministic Python downstream of LLM extraction. Tests inject simulated extraction JSON via `--fields-json`.

7. **Each test runs in isolation** — uses `env_dir` (tmp_path) so leads.json starts empty. Pre-seeded leads where needed (e.g., C04) are written via direct JSON file write before the test invocation.

Each test follows: `test_c<NN>_<snake_case_summary>`. Examples:
- `test_c01_clean_unknown_sender_creates_lead`
- `test_c03_staff_referral_routes_to_friend_not_staff`
- `test_c20_prompt_injection_shaped_notes_no_crash`

### Test structure template

```python
def test_c<NN>_<summary>(env_dir, bridge_server):
    """v3.1 C<NN> — <short doc-spec summary>.

    Doc-spec assertions (from docs/catering-edge-cases.md):
    - <bullet 1>
    - <bullet 2>
    - <bullet N>
    """
    port, BridgeStub = bridge_server
    BridgeStub.requests.clear()

    fields = {<simulated extraction JSON>}
    r = _run_create(env_dir, port, fields, message_id="MSG_C<NN>_001")
    assert r.returncode == 0

    # Assertions matching the doc-spec bullets, in order
    leads_path = env_dir / "state" / "catering-leads.json"
    leads = json.loads(leads_path.read_text())
    lead = leads["leads"][0]
    assert lead["<field>"] == <expected>
    # ...
```

## Risks + open questions

1. **Test concentration in one file.** `tests/test_catering_v02_scripts.py` already has 32 tests; adding 18 brings it to 50 tests in one file (~1500 LOC). Consider splitting into `tests/test_catering_v02_b1_cases.py` for v3.1 doc-spec tests vs `test_catering_v02_scripts.py` for development-driven tests. **Decision deferred to design phase** — splitting is cleaner but adds a new test file to maintain.

2. **C04's negative assertion is weak.** "Lead's customer_phone is unknown phone, NOT Priya's" — what if the `create-catering-lead` script silently uses sender_phone as customer_phone (current behavior)? Then the test passes by accident: the unknown phone IS the sender_phone, so the assertion holds even without any lookup logic. The privacy invariant the case is testing is "no automatic linking via notes content" — but `create-catering-lead` doesn't do that anyway. The test becomes a tautology against current behavior. **Mitigation:** explicitly assert that running `lookup_prior_leads_by_phone(unknown_phone)` returns empty AND `create-catering-lead` doesn't peek at the lookup result from the notes parsing. Both naturally true today.

3. **C16 + C17 test apply-catering-owner-decision** which has its own existing tests (`test_apply_approve_*`). Adding 2 more test cases for menu-filter logic specifically. The script's `_load_menu_filtered` function is the actual unit; a unit test against that helper is cleaner than the full apply-flow integration test. **Decision deferred to design phase.**

4. **C20 prompt-injection test asserts "no crash."** Boundary: the test passes if the script exits successfully OR exits with EXIT_INVALID_INPUT (Pydantic rejection). Both outcomes are acceptable per the case spec. Need to assert no Python exception traceback in stderr — use `r.returncode in (0, 2)` and `"Traceback" not in r.stderr`.

5. **Pre-seeding leads for C04.** Test must write `catering-leads.json` with Priya's existing leads BEFORE running create-catering-lead. Use the same `_seed_leads` helper from `test_lookup_prior_leads.py` (or duplicate inline; ~10 LOC).

6. **C19 status transition.** The `CateringLeadStatusChange` audit entry has `from_status="NEW"` and `to_status="AWAITING_OWNER_APPROVAL"` per `create-catering-lead:373-377`. Existing test #221 implicitly verifies this via `len(BridgeStub.requests) == 1` (card sent). Explicit C19 test pins the audit entry's `from_status` and `to_status` fields directly via `decisions.log` parsing.
