### Agent #3 Multi-Location Coordinator — FAIL → HOTFIX

**Test 1 (empty locations, expected exit-2):** PASS
```
$ closest-location.py --address "Houston TX" --top-n 3
{"source": "not_configured", "results": [], "n_locations_total": 0, ...}
```

**Test 2 (3-location fixture + real address geocode):** FAIL
```
$ closest-location.py --address "4500 Westheimer Rd Houston TX" --top-n 3 ...
could not geocode address: '4500 Westheimer Rd Houston TX'
```

**Root cause: BUG E2E-2** — closest-location.py invokes `maps_client.py search --query "X" --limit 1` but the deployed maps_client.py CLI expects `search "X"` (positional only). The flag-based call always returns "unrecognized arguments". Verified directly:
```
$ maps_client.py search --help
usage: maps_client.py search [-h] query [query ...]
```

**Root cause: BUG E2E-3** — same script's `osrm_distance()` calls `maps_client.py distance --from-lat ... --from-lon ... --to-lat ... --to-lon ...`, but the actual CLI is `distance "Origin Address" --to "Dest Address"` (addresses, not lat/lon). The OSRM-routing path was never reachable in v0.1.

**Drift-rule failure:** I assumed the maps_client API without verifying. Unit tests at `tests/test_agent_3_multi_location.py` mocked the subprocess output rather than calling it for real, so neither bug surfaced in CI. Same class as the load_model-vs-yaml.safe_load bug from PR #64.

**Hotfix (this PR):**
- `geocode_address()`: switch to positional arg `[..., "search", address]`. Updated to parse the actual response shape (single object with `latitude`/`longitude` keys, not a `results` list).
- `osrm_distance()`: returns None unconditionally in v0.1 → triggers Haversine fallback (urban-detour-factor 1.3, urban km/min 0.5). v0.2 path documented in code: call OSRM HTTP directly OR extend maps_client.py to accept lat/lon. The fallback is sufficient for "which of N stores is closest?" — the precision improvement from real OSRM-routing was nice-to-have.

### Agent #4 Daily Brief — FAIL → HOTFIX

**Test:** `send-daily-brief --force --dry-run`

```
TypeError: can't compare offset-naive and offset-aware datetimes
  File "/opt/shift-agent/log_source.py", line 91, in iter_entries
    if start_ts <= ts < end_ts:
```

**Root cause: BUG E2E-1** — log_source.py:87 parses ts with `datetime.fromisoformat(ts_str.replace("Z", "+00:00"))`. If a log entry has a NAIVE timestamp (no Z, no +00:00), the parsed datetime has `tzinfo=None`. The comparison at line 91 against tz-aware `start_ts`/`end_ts` raises TypeError.

**Verified:** 1 naive entry in production audit log (`2026-05-03T16:07:10.473035`) out of 196 total. Likely written by a script before the v0.3 _BaseEntry tz-aware-only invariant landed. The `_BaseEntry` validator (schemas.py:1338-1359) auto-promotes naive→UTC at READ time; log_source.py needed the same defense.

**Hotfix (this PR):** `if ts.tzinfo is None: ts = ts.replace(tzinfo=timezone.utc)` before the comparison. Mirrors `_BaseEntry._ensure_tz_aware` validator behavior.

## Systemic finding: deploy chicken-and-egg

**Symptom:** After PR #66 deploy completed (smoke-test green), 20 of 52 SKILLs were missing from `/root/.hermes/skills/`. Specifically: `handle_sick_call`, all `catering_*`, `compliance_owner_query`, `expense_bookkeeper_dispatcher`, `pnl_anomaly_dispatcher`, `equipment_maintenance_dispatcher`, etc.

**Root cause:** The `install_artifacts()` function in `shift-agent-deploy.sh` is loaded into the running shell's memory. When the function runs `install -m 755 src/agents/shift/scripts/* /usr/local/bin/`, it OVERWRITES `/usr/local/bin/shift-agent-deploy.sh` with the new version. But the running shell continues using the OLD in-memory function for the rest of THIS deploy. New install rules added to deploy.sh (multi_location, compliance, equipment_maintenance, pnl_anomaly, etc.) silently skip until a SECOND deploy runs with the new in-memory function.

**Found by:** Phase 1 inventory revealed only 6 of expected 26 per-agent SKILLs visible. Re-running the deploy from the now-installed-on-disk new deploy.sh restored all 52 SKILLs.

**Smoke test gap:** the smoke test probes scripts via `--help`, not SKILLs. SKILL-presence is silent at deploy time. Worth adding a smoke-test probe: `[ -d /root/.hermes/skills/<expected-skill> ]` for each agent's expected skill.

**Action item (deferred):** harden `shift-agent-deploy.sh` to either (a) re-exec itself after self-overwrite to pick up the new in-memory function in a single pass, OR (b) add a SKILL-presence probe to the smoke test that fails the deploy if expected SKILLs are missing.
