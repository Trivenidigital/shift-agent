# Catering Agent 100% Autonomy + Self-Learning Rails Plan

**Drift-check tag:** extends-Hermes

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` to implement this plan task-by-task. Keep the runtime self-learning boundary from `AGENTS.md`: production agents may update state, memory, reports, and owner-visible suggestions; code, SKILL, prompt, and deploy-config changes require tests, review, PR, and deploy.

**Goal:** Reach a WhatsApp-first Catering Agent that behaves like a strong restaurant catering coordinator: captures leads, updates menus from owner media, proposes grounded menus, remembers useful history, prepares owner approval cards, follows up, and improves from evidence while preserving final owner approval.

**Pushback:** Treat "100%" as the destination architecture, not one giant production PR. The safe path is a series of small, reviewed production slices. The first build slice in this cycle is owner-visible catering learning generated by the existing pattern-report job and rendered by the Daily Brief, because it uses existing Hermes/state primitives, is read-only, and immediately helps the owner improve menus and operations.

## 100% Destination Capabilities

1. **Lead capture and qualification**
   - Any WhatsApp sender can create a catering inquiry, including employees asking for their own family events.
   - Extract event date, headcount, serving time, delivery/pickup, diet mix, off-menu requests, venue, urgency, and contact details.
   - Never ask for required metadata that was already present in message metadata.

2. **Menu authority and menu freshness**
   - Owner or verified employee can submit image/PDF/text menu sources.
   - Owner applies menu updates by code.
   - Daily Brief shows menu freshness and top requested off-menu items so the owner knows what to update next.

3. **Grounded proposals**
   - Customer asks for one, two, or three proposal menus.
   - System generates menu-grounded options from uploaded menu only.
   - No customer-facing pricing, deposit, or booking confirmation before owner approval.

4. **Selection and owner approval**
   - Customer selects option number/title/tier unambiguously.
   - System persists selection, prepares owner approval card, and sends only after owner `#CODE approve`.
   - Bridge failures and stale selections are handled without lying to the customer.

5. **Event packet / BEO-lite**
   - After owner approval, generate a compact event packet: menu, quantities, prep notes, delivery address/time, staffing/packing checklist, and customer communications history.

6. **Production kitchen support**
   - Aggregate by date: production list, pack sheet, dietary/allergen notes, pickup/delivery schedule, unresolved fields.
   - Capacity guard: warn owner if date/headcount collides with configured max events or prep capacity.

7. **Self-learning with rails**
   - Runtime learning: derive read-only insights from leads/proposals/menu state.
   - Operator learning: append findings to owner-visible reports and backlog.
   - Code/SKILL learning: use Hermes Self-Evolution Kit offline with eval datasets, then PR/review/deploy.

8. **Business growth loop**
   - Track conversion by status, proposal request rate, selected tiers, off-menu demand, repeat-customer hints, and common missing fields.
   - Suggest menu/package improvements, not silently change production policy.

## First Build Slice: Catering Pattern Learning Summary In Daily Brief

**Why this slice:** It moves toward 100% autonomy without changing customer behavior. It turns real WhatsApp traffic into owner-visible business learning: off-menu demand, missing-field friction, proposal health, and menu freshness. Scope-correct implementation extends `catering-pattern-report`; Daily Brief only consumes a sanitized readout so `send-daily-brief` does not become a second learning engine.

### First-slice Hermes/net-new step tags

| Step | Hermes/repo capability? | Decision |
|---|---|---|
| Read catering leads/menu/proposals | `[Hermes]` JSON state + Pydantic schemas + existing safe load patterns | Use existing state files and schema fields. |
| Nightly learning/report job | `[Hermes]` existing `catering-pattern-report.timer` + script | Extend this script instead of adding a new aggregator. |
| Daily owner delivery | `[Hermes]` existing `send-daily-brief.timer` + template | Render only sanitized learning summary. |
| Sanitized learning summary JSON | `[net-new]` | Add bounded sidecar generated by `catering-pattern-report`. |
| Free-text safety | `[net-new]` | v1 stores/render counts only for off-menu asks; no customer-derived terms. |
| Kill switch / opt-in | `[net-new]` | Gate Daily Brief learning lines separately from `cfg.catering.enabled`. |

**Files:**

- Modify: `src/agents/catering/scripts/catering-pattern-report`
- Modify: `src/agents/daily_brief/scripts/send-daily-brief`
- Add: `tests/test_daily_brief_catering_learning.py`
- Add or extend: `tests/test_catering_pattern_report.py`
- Modify: `docs/runbooks/production-pilot-shift-catering-daily-brief.md`
- Modify: `tasks/todo.md`

**Behavior:**

- Extend `catering-pattern-report` to write a sanitized sidecar such as `/opt/shift-agent/state/catering-learning-summary.json`.
- Extend the existing `*Catering pipeline:*` block to read that sidecar only when the learning section is enabled.
- Summary fields:
  - proposal health over the last 30 days: sent, selected, send_failed, select_failed
  - menu learning over the last 30 days: off-menu request count and lead count
  - active missing-info count: active leads missing headcount or event date
  - menu freshness: days since `Menu.updated_at`, not filesystem mtime
- Do not include customer names, phone numbers, prices, totals, deposits, payment rails, venue addresses, or raw inquiry text.
- Do not read or summarize `CateringProposalSet.request_text`; it is raw customer free text.
- Do not persist or render raw/sanitized `off_menu_items` text in v1; counts only until an owner-curated food taxonomy exists.
- If state files are missing, corrupt, schema-invalid, unexpected shape, or unreadable, contain the error inside the catering helper and return the existing safe catering block or omit learning lines. The Daily Brief must not fail because learning state is bad.
- Add module-level path constants or env overrides for `CATERING_LEADS_PATH`, `CATERING_PROPOSALS_PATH`, `CATERING_MENU_PATH`, and `CATERING_LEARNING_SUMMARY_PATH` so tests never touch real `/opt` state.
- Add an opt-in/kill switch independent of `cfg.catering.enabled`; default off until `send-daily-brief --dry-run` output is reviewed on the VPS.

**Tests:**

- Active lead with off-menu requests renders only counts.
- Customer names, phones, addresses, dates, prices, raw inquiry text, raw off-menu text, and proposal `request_text` do not appear in the learning lines.
- Missing/corrupt/schema-invalid lead/proposal/menu/summary state degrades without crashing the brief.
- Proposal status counts render only for the last 30 days and use exact buckets: `SENT` and `SUPERSEDED` count as offered/sent, `SELECTED` counts selected, `SEND_FAILED` counts send_failed, `SELECT_FAILED` and `SELECTED_OWNER_CARD_FAILED` count select_failed. Older rows are ignored.
- Learning lines are absent when the opt-in switch is disabled.

## Later Slices

1. **Menu update acceptance pack**
   - Live smoke owner media upload -> pending menu -> owner apply -> customer proposal uses new menu.
   - Add fixture-based test for generic captions: `menu`, `new menu`, `update menu`.

2. **Catering lead completeness helper**
   - Owner-only report of unresolved date/headcount/time/location for active leads.
   - Optional customer follow-up suggestions, but owner-approved before sending.

3. **BEO-lite event packet**
   - Deterministic packet after owner approval.
   - No customer-facing price math beyond approved quote.

4. **Capacity guard**
   - Configurable per-day max events/headcount.
   - Owner alert when a new inquiry exceeds capacity or conflicts with existing approved events.

5. **Offline Self-Evolution Kit loop**
   - Export anonymized eval cases from decisions.log and state.
   - Run self-evolution against SKILL candidates offline.
   - Promote only via tests, reviewers, PR, deploy.

6. **MCP integrations**
   - Check `mcp/native-mcp` before custom integrations.
   - Likely candidates: calendar, documents, Sheets/Airtable, e-sign, payment links, QBO.
